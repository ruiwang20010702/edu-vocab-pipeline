## 1. 项目的核心矛盾

### 矛盾表述

**「AI 生成的规模化效率」与「教育内容零容错的质量要求」之间的根本张力。**

展开来说：

- **效率端**：2557 个词 × 3 个内容维度 = ~8000 条内容，如果全部人工编写，按每条 3 分钟计算需要 400 小时（约 50 个工作日）。AI 生成可以在 15 秒内完成规则引擎版本，或 10 分钟内完成 AI 版本。效率提升约 2000 倍。

- **质量端**：最终用户是中小学生。一条错误的例句（比如"She is a good student"配上"货物"的释义）不仅无法帮助学习，还会造成混淆。教育产品的容错率本质上是零——你不能对学生说"大部分是对的"。

### 这个矛盾决定了什么

整个系统的架构都是这个矛盾的展开。你会在后续每一个设计决策中看到同一个模式：**先用机器做规模化生产，再用多层过滤确保输出质量，最终以人工审核兜底**。这不是因为我们不信任 AI，而是因为教育内容的错误成本远高于漏掉一条好内容的机会成本。

用质量管理的术语说，我们选择的是 **fail-safe**（宁可多拦截，不可放过一条错的）而非 **fail-open**（宁可放过，不可误杀）。

### 被否决的另一条路

**全人工路线**：学科产品直接在 Excel 中编写所有内容，不用 AI。

否决原因：
- 400 小时的人工成本不可接受
- 人工编写同样无法保证零错误率——人类犯的错误类型不同，但不会更少
- 一旦教材改版或标准调整，全量重做的边际成本与首次相同

**全自动路线**：AI 生成 + AI 审核，完全无人干预。

否决原因：
- 2024-2025 年的 LLM 在细粒度语义判断上（例如"这条例句是否体现了 good 作为名词'货物'的义项"）的准确率大约 85-92%，对教育产品来说不够
- 自动审核无法处理标准本身的模糊地带——"a kind of"作为语块是太简单了还是恰到好处？这需要学科判断
- 教育监管对 AI 生成内容有审核要求，全自动路线存在合规风险

**我们选择的中间路线**：AI 规模化生产 + 规则引擎零成本过滤 + AI 语义审核 + 人工兜底。四道漏斗逐层收窄，每层只做自己最擅长的事。

## 2. 六条设计原则的方法论溯源

### 原则一：标准前置

**方法论来源**：ISO 9001 质量管理体系 + 教育课程标准制定流程

**为什么要这么做**

在没有标准的情况下，"好"是主观的。一个学科产品认为"run fast"是好语块，另一个可能认为"run quickly"更好。如果生产 Prompt 和审核 Prompt 各自"理解"标准，就会出现"生产时按 A 标准做，审核时按 B 标准判"的不一致。

标准前置的意思是：**先有标准，再有生产，最后有审核**，且三者都从同一个机器可读文件 `standards.yaml` 中派生。

**具体落地**

```yaml
# standards.yaml 是唯一仲裁源
content:
  chunk:
    constraints:
      must_contain_target_word: true    # 规则引擎和 AI 生成器都从这里读取约束
    evaluation:
      criteria:
        - name: "自然度"
          weight: 0.4                   # AI 审核器从这里读取评分权重
```

生产 Prompt、审核 Prompt、规则校验逻辑，都从这个文件中读取参数。改了标准文件，三者同步变化。

**优点**

| 维度 | 分析 |
|------|------|
| 一致性 | 生产和审核看同一份标准，消除"标准漂移"问题 |
| 可追溯 | 标准变更有 Git 历史，可以回溯任何一次调整的原因 |
| 机器可读 | 规则引擎可以直接从 YAML 文件中提取约束参数，而不是硬编码在代码中 |
| 协作友好 | 学科产品修改 YAML 中的正反例，不需要改代码 |

**缺点与局限**

| 维度 | 分析 |
|------|------|
| 学习门槛 | 学科产品需要学习 YAML 语法——虽然很简单，但比 Word 文档多了一层障碍 |
| 表达力有限 | YAML 难以表达复杂的语义约束（如"例句语境应贴近学生日常生活"），这类约束只能写在文字描述中，机器无法直接执行 |
| 标准滞后 | 标准的制定依赖学科产品，如果标准迟迟不出（当前正是这个状态——`standards.yaml` 中很多字段只有框架没有内容），整个生产流程就卡在第一步 |

**被否决的替代方案**

- **Word 文档 + 人工翻译成代码**：标准由学科产品用 Word 写，AI 产品经理翻译成代码。否决原因：翻译环节引入理解偏差，标准更新时需要重新翻译。
- **数据库存储标准**：标准存在数据库表中。否决原因：过度工程化——当前只有一个标准文件，且变更频率低（预计每月 1-2 次），用 Git 版本控制即可。
- **JSON 格式**：技术上等价，但 JSON 不支持注释，学科产品无法在标准旁边写备注。YAML 的注释能力是关键差异。

### 原则二：事实与生成分离

**方法论来源**：关注点分离（Separation of Concerns）+ 数据仓库中的事实表 vs 维度表

**为什么要这么做**

一个单词的音标和释义是**客观事实**——不管你用什么 AI 模型、什么 Prompt，"kind"的音标都是 `/kaɪnd/`，"友好的"就是"友好的"。

但一个单词的语块、例句、助记是**生成内容**——不同的模型、不同的 Prompt 版本、不同的生成时间，会产生不同的结果。生成内容需要审核、可能被驳回、可能重新生成。

如果把事实和生成内容混在一张表里：
- 重新生成语块时，可能误改音标
- 审核例句时，不知道这条例句是用什么模型在什么时间生成的
- 一次生成失败不应该影响已有的事实数据

**具体落地**

```
Layer 1（数据层）：words + phonetics + meanings + sources
    ↓ 只读关系
Layer 2（内容层）：content_items（word_id + meaning_id + dimension → content + status）
```

数据层只存客观事实，不可被内容生成流程修改。内容层的每一条记录都标注了生成模型 `model`、Prompt 版本 `prompt_version`、审核状态 `status`。

**优点**

| 维度 | 分析 |
|------|------|
| 安全性 | 重新生成内容不会破坏事实数据 |
| 可重跑性 | 可以安全地对所有词重新生成语块，不影响音标和释义 |
| 审计能力 | 每条内容记录了生成模型和 Prompt 版本，可以追溯质量问题的根因 |
| 独立演进 | 数据层和内容层可以独立迭代——换模型只影响内容层，换词典只影响数据层 |

**缺点与局限**

| 维度 | 分析 |
|------|------|
| 查询复杂度 | 组装完整数据需要 JOIN 多张表。已通过 ORM eager loading 优化 |
| 概念负担 | 新接手的开发者需要理解"为什么语块不直接存在 word 表上"——这个设计的合理性需要上下文才能理解 |

### 原则三：单一职责

**方法论来源**：SOLID 原则中的 SRP（Single Responsibility Principle）

**为什么要这么做**

三个内容维度（语块、例句、助记）的生成逻辑完全不同：

| 维度 | 输入 | 输出 | 核心难点 |
|------|------|------|---------|
| 语块 | 词+义项 | 一个短语 | 搭配的自然度和高频性 |
| 例句 | 词+义项 | 英文句子+中文翻译 | 义项匹配+难度控制+翻译准确 |
| 助记 | 词+音标 | 记忆技巧 | 创造性+准确性+趣味性 |

如果把三者放在一个生成器中，任何一个维度的 Prompt 调整都会影响其他两个。拆分后，`ChunkGenerator` 只关心语块，迭代 Prompt 时不需要担心例句受影响。

同理，质量层的 10 项检查各自独立——"音标格式检查"的逻辑不应该和"语块包含目标词检查"混在一起。

**优点**：修改一个维度时认知负担最小，测试可以只跑受影响的生成器。

**缺点**：文件数量更多（3 个生成器文件 vs 1 个），但每个文件都短小（~100 行），整体可维护性更好。

---

### 原则四：原子操作

**方法论来源**：数据库事务原子性 + Unix 哲学（做一件事，做好它）

**为什么要这么做**

CLI 支持对单个词的单个维度执行操作：

```bash
python cli.py generate sentence kind    # 只重新生成 kind 的例句
python cli.py reject 42 "难度过高"       # 只驳回一条内容
```

这意味着：
- 学科产品发现 kind 的例句不对，可以只重新生成 kind 的例句，不影响 kind 的语块和助记，更不影响其他词
- 驳回一条内容后重新生成，只操作那一条
- 生成中途失败（比如 API 超时），已经成功的部分不受影响

**优点**：精确控制 + 错误隔离。

**缺点**：对单个词做操作时，不能自动级联——比如改了 kind 的释义后，语块和例句不会自动重新生成。这是有意为之的（自动级联可能意外覆盖已审核通过的内容），但需要操作人员理解这个行为。

---

### 原则五：质量门禁

**方法论来源**：制造业 Quality Gate + 软件工程 CI/CD Pipeline

**为什么要这么做**

教育内容的错误成本远高于拦截成本。一条错误的例句进入学生端，影响面可能是数千个学生。而一条好内容被门禁拦住，最多延迟一个审核周期。

```
Gate 1 → Gate 2 → Gate 3 → Gate Export
                                ↓
                           fail-safe：
                    未通过的不放行，而非
                    默认放行再事后检查
```

关键设计：**导出门禁默认阻断**。`python cli.py export` 在有未审核内容时会拒绝导出。必须用 `--force` 才能强制导出（仅限调试用途）。

**优点**：从流程上杜绝"忘了审核就上线"的可能。

**缺点**：开发调试时每次都要加 `--force`，有轻微摩擦。但摩擦是有意设计的——它提醒操作者"这不是正式环境应有的操作"。

---

### 原则六：审计可追溯

**方法论来源**：金融/医疗监管的审计链 + GDPR 数据处理记录

**为什么要这么做**

当一条例句出了问题，需要回答三个问题：
1. 这条例句是谁生成的？（哪个模型、哪个 Prompt 版本）
2. 是谁审核通过的？（自动通过还是人工通过）
3. 什么时候通过的？

没有审计日志，这三个问题都无法回答。

**具体落地**

```
content_items 表：model + prompt_version + status + reviewer + reviewed_at
audit_log 表：action + actor + old_value + new_value + created_at
```

每次 `generate`、`approve`、`reject` 操作都会写入 `audit_log`。

**优点**：完整的操作链条可追溯，方便定位质量问题的根因。

**缺点**：`audit_log` 会随操作量线性增长。当前数据规模下（~10000 条操作记录）完全可接受。如果未来数据量达到百万级，需要考虑归档策略。

## 3. 关键架构决策与权衡

### 决策一：四层架构而非三层

**我们的选择**：标准层（L0）→ 数据层（L1）→ 内容层（L2）→ 质量层（L3）→ 交付层（L4）

**为什么不是经典三层（数据-业务-展示）**

传统三层架构关注的是"请求如何流经系统"，但我们的核心问题不是请求处理，而是"内容如何从原始数据变成可信的学习材料"。四层架构反映的是**内容的成熟度**：

| 层级 | 内容成熟度 | 类比 |
|------|-----------|------|
| L0 标准层 | 规范——还没有内容，只有"好内容长什么样"的定义 | 工厂的质量标准手册 |
| L1 数据层 | 原料——有了单词、音标、释义，但还没有语块和例句 | 工厂的原材料仓库 |
| L2 内容层 | 半成品——语块和例句已生成，但还没审核 | 流水线上的产品 |
| L3 质量层 | 检验——通过规则检查和人工审核后标记为 approved | 质检合格贴标 |
| L4 交付层 | 成品——只有 approved 的内容才能导出给前端 | 出厂发货 |

**被否决的替代方案**

- **两层（数据+应用）**：把所有逻辑放在一个应用层中。否决原因：无法独立迭代生成和审核——改了生成逻辑需要同时考虑审核的影响。
- **微服务**：每层一个独立服务。否决原因：当前团队规模（1-3 人）和数据规模（2557 词）不支持微服务的运维复杂度。单体应用完全够用。

### 决策二：CLI 驱动而非 Web UI

**我们的选择**：所有操作通过 `python cli.py <command>` 执行

**为什么不做 Web UI**

| 对比维度 | CLI | Web UI |
|---------|-----|--------|
| 开发成本 | ~2 天 | ~2-4 周（前后端 + 部署） |
| 操作可重复性 | 命令可复制粘贴、写入脚本 | 需要录屏或写操作文档 |
| 批量操作 | 天然支持脚本化 | 需要额外开发批量操作界面 |
| 审计能力 | 命令行历史天然记录操作 | 需要额外实现操作日志 |
| 协作体验 | 不友好——学科产品需要学命令行 | 友好——点击式操作 |
| 实时反馈 | 终端输出，即时可见 | 需要做加载状态、进度条 |

**当前阶段的取舍**：项目处于"系统搭建"阶段，操作者主要是 AI 产品经理和开发者。CLI 的开发效率优势（2 天 vs 4 周）决定了它是当前最优选择。

**未来演进**：当学科产品需要自主审核时（P2 优先级），需要在 CLI 之上加一层 Web UI。推荐方案是 Streamlit 或 Gradio——用 Python 直接包装现有函数，无需写前端代码，半天可完成。

### 决策三：ORM (SQLAlchemy) 而非原始 SQL

**我们的选择**：SQLAlchemy 2.0 声明式 ORM

**为什么从原始 SQL 迁移到 ORM**

| 对比维度 | 原始 SQL | SQLAlchemy ORM |
|---------|---------|----------------|
| 数据库可移植性 | 锁死在 SQLite（用了大量 SQLite 方言语法如 `INSERT OR IGNORE`、`datetime('now')`） | 同一套代码自动适配 SQLite 和 PostgreSQL |
| 类型安全 | SQL 字符串拼接，IDE 无法检查语法错误 | ORM 类有类型提示，IDE 可做自动补全和错误检查 |
| N+1 问题 | 手动 JOIN，容易写出 N+1 查询 | `joinedload` / `selectinload` 一行解决 |
| 代码可读性 | 134 行 DDL 字符串 + 24 处 raw SQL | 170 行模型定义，业务逻辑读起来像英语 |
| 迁移能力 | 无（每次改表手动写 ALTER TABLE） | 支持 Alembic 自动生成迁移脚本（当前未引入，零成本可加） |

**优点**

- `export.py` 从 ~10000 次查询（每个词 4 次查询 × 2557 词）优化到 ~4 次查询，靠的就是 ORM 的 eager loading
- `data_layer.py` 的 24 处 `INSERT OR IGNORE` / `ON CONFLICT` 统一为"先查后写"模式，跨库兼容
- 模型定义即文档——看 `core/models.py` 就能理解数据结构，不需要另外维护 ER 图

**缺点**

- 团队需要熟悉 SQLAlchemy API（学习成本约 1-2 天）
- ORM 查询在极端复杂场景下可能不如手写 SQL 灵活（但当前项目的查询复杂度远未触及此上限）
- "先查后写"比 `INSERT OR IGNORE` 多一次 SELECT，但在 2557 词的规模下性能差异可忽略（毫秒级）

## 4. 数据模型设计的推演过程

### 核心问题：内容应该挂在哪里？

这是整个数据模型设计中最关键的决策。有三种方案：

#### 方案 A：内容挂在单词上

```
words → content_items (word_id, dimension, content)
```

**优点**：最简单，一个 word_id 对应一条语块、一条例句。

**致命缺陷**：多义词。`good` 有"好的"（adj.）和"货物"（n.）两个义项。如果语块只挂在单词上，"be good at" 和 "a kind of goods" 都关联到 `good`，无法区分哪个语块对应哪个义项。导出给前端时，前端也无法把语块和义项对应起来。

**结论**：否决。

#### 方案 B：内容全部挂在义项上

```
meanings → content_items (meaning_id, dimension, content)
```

**优点**：解决了多义词问题。每个义项有自己的语块和例句。

**缺陷**：助记是面向整个单词拼写/发音的（"kind 和 king 只差一个字母"），与义项无关。如果助记也挂在义项上，`good` 的两个义项各生成一条助记，但内容可能完全相同——浪费且不合逻辑。

**结论**：助记的粒度与语块/例句不同，不能一刀切。

#### 方案 C（我们的选择）：混合粒度

```
content_items (word_id, meaning_id, dimension, content)
  - chunk/sentence：meaning_id 指向具体义项（必填）
  - mnemonic：meaning_id 为 NULL（词级）
```

用 `UNIQUE(word_id, meaning_id, dimension)` 约束保证：
- 每个义项的每个维度只有一条内容
- `meaning_id = NULL` 时，每个词只有一条助记

**优点**：精确反映业务实际——语块和例句是义项级的，助记是词级的。

**缺点**：NULL 外键在某些 ORM 场景下需要特殊处理（例如查询时需要区分 `meaning_id IS NULL` 和 `meaning_id = 某个值`）。但 SQLAlchemy 处理 NULL 比较自然，实践中没有问题。

### 推演过程的关键洞察

**"义项是一切内容的锚点"**这个认识不是一开始就有的。它来自一个真实的错误案例：

> 早期测试中，`good` 生成的例句 "She is a good student." 被标注在"货物"义项下。这条例句本身没错，但配上"货物"就成了张冠李戴。
>
> 根因分析：content 只关联到 word，没有关联到 meaning。生成器无法区分哪个义项需要什么样的例句。

这个 bug 直接催生了"内容挂在义项上"的设计。

### 9 张表的设计决策明细

| 表 | 为什么需要这张表 | 被否决的替代方案 |
|-----|----------------|----------------|
| `words` | 单词去重后的唯一标识。ID 从 1001 起，避免与前端已有 ID 冲突 | ID 自增——否决，因为需要与前端约定起始值 |
| `phonetics` | 音标和音节独立存储，与 words 1:1。独立的原因是音标有自己的数据源（`eng_to_ipa`）和更新逻辑 | 直接放在 words 表中——否决，音标数据来源可能变更（从本地库换成 API），独立表方便替换 |
| `meanings` | 义项级存储，UNIQUE(word_id, pos, definition)。同一个词的不同义项需要分开记录 | JSON 数组存在 words 表中——否决，无法做外键关联，无法按义项做精确操作 |
| `sources` | 来源教材追踪，UNIQUE(meaning_id, textbook)。同一义项可来自多份教材 | 用逗号分隔的字符串存在 meanings 表中——否决，无法精确查询"某教材包含了哪些义项" |
| `content_items` | 生成内容的统一存储，UNIQUE(word_id, meaning_id, dimension) | 每个维度一张表（chunks 表、sentences 表、mnemonics 表）——否决，三张表的 schema 几乎相同，分开只增加维护成本 |
| `quality_issues` | 质量问题记录，resolved 标记是否已解决 | 不持久化，每次检查实时计算——否决，需要追踪"这个问题什么时候被发现、什么时候被解决" |
| `audit_log` | 操作日志，append-only | 用文件日志代替——否决，数据库日志可查询、可关联到具体单词 |
| `packages` | 词包定义。当前只有一个，但架构上支持多个 | 不建表，导出时按条件过滤——否决，词包有自己的属性（名称、目标受众），需要独立管理 |
| `package_meanings` | 词包-义项关联，多对多关系 | 在 meanings 表加 package_id 列——否决，一个义项可以属于多个词包，多对多关系需要中间表 |

## 5. 三层质量漏斗的 ROI 分析

### 漏斗模型

```
   原始内容池（~8000 条）
        ↓
   Layer 1: 规则校验（10 项检查）
        ↓ 过滤掉格式错误（~76 条）
   Layer 2: AI 语义审核（6 项检查）
        ↓ 过滤掉语义错误（预估 ~200-400 条）
   Layer 3: 人工审核
        ↓ 过滤掉主观不达标（预估 ~100-200 条）
   最终产出（~7000+ 条 approved 内容）
```

### 每一层的 ROI 分析

#### Layer 1：规则校验（已实现）

| 指标 | 数值 |
|------|------|
| 成本 | 零（纯代码逻辑，秒级运行） |
| 发现问题数 | 76 条 |
| 发现的问题类型 | 格式问题（音标缺失、例句不含目标词、缺翻译等） |
| 准确率 | 100%（规则检查是确定性的） |
| 漏检率 | 高——规则无法检查语义问题（如"例句是否体现了正确的义项"） |

**ROI 分析**：开发成本约 1 天，可以永久性地自动发现所有格式类问题。这一层的投入产出比极高，任何项目都应该首先实现它。

**关键认识**：规则校验能发现的问题都是"确定性错误"——要么对、要么错、没有灰色地带。这类问题交给人工审核是浪费人力。

#### Layer 2：AI 语义审核（未实现，6 个检查器）

| 指标 | 预估数值 |
|------|---------|
| 成本 | ~$2-5（使用 Claude Haiku / GPT-4o-mini） |
| 预估发现问题数 | 200-400 条 |
| 问题类型 | 语义错误（张冠李戴、难度不匹配、搭配不自然等） |
| 预估准确率 | 85-92% |
| 预估漏检率 | 8-15% |

**ROI 分析**：$5 成本能过滤掉 200-400 条有语义问题的内容。如果这些内容全部进入人工审核，按每条 3 分钟计算，需要 10-20 小时人力。AI 审核的 ROI 约为 2-4 小时/$1。

**当前状态：未实现。** 这是因为 AI 审核的标准（"什么算难度过高？""什么算搭配不自然？"）需要学科产品定义，而学科产品尚未产出审核标准。这是项目当前最大的阻断项。

**优先级排序**（按投入产出比）：

| 检查器 | 优先级 | 原因 |
|--------|--------|------|
| 例句-释义一致性 | P0 | 张冠李戴是最严重的错误类型，且 LLM 判断准确率高（~95%） |
| 例句难度适配 | P1 | 难度不匹配会导致学生看不懂或觉得太简单 |
| 语块自然度 | P1 | 不自然的搭配会误导学生 |
| 助记有效性 | P2 | "助记只是复述释义"是常见问题，但不会造成错误 |
| 词性正确性 | P2 | 词性错误已在规则层部分覆盖 |
| 同义释义检测 | P2 | 检测"同一含义被当成两个义项"，边界案例较多，准确率可能不高 |

#### Layer 3：人工审核

| 指标 | 预估数值 |
|------|---------|
| 成本 | 10-15 小时学科产品人力 |
| 处理问题数 | ~200-300 条（AI 审核标记为低置信度的 + AI 审核漏掉的） |
| 准确率 | ~98%（人也会犯错，但错误率远低于 AI） |
| 能处理的问题类型 | 所有——包括规则和 AI 都无法处理的主观判断 |

**ROI 分析**：人力成本最高、处理速度最慢，但能处理最复杂的问题。三层漏斗的设计就是为了**把人力聚焦在只有人才能判断的问题上**——格式问题交给规则，语义问题交给 AI，只有"这条助记够不够有趣""这个搭配适不适合初中生"这类需要学科判断的问题才进入人工审核。

### 三层漏斗的经济学

```
规则校验：     $0 → 发现 76 条（格式错误全覆盖）
AI 语义审核：  $5 → 发现 ~300 条（语义错误大部分覆盖）
人工审核：     15h → 处理 ~200 条（剩余的灰色地带）

对比：如果没有前两层，人工需要审核全部 8000 条 → ~400 小时
实际人工审核量：~200 条 → ~15 小时

前两层为人工审核节省了 96% 的工作量。
```

## 6. 双模式生成器的产品逻辑

### 为什么需要规则引擎

每个生成器支持两种模式：`rule_engine`（零成本、确定性）和 AI 模型（有成本、高质量）。

**规则引擎不是"穷人的 AI"。** 它解决的是一组不同的问题：

| 场景 | 规则引擎 | AI 模型 |
|------|---------|--------|
| 开发调试 | 15 秒跑完全量，验证流水线逻辑 | 10 分钟 + $3-8，调试流水线不需要这么高质量的内容 |
| API 不可用 | 系统不停摆 | 报错，流程中断 |
| 成本控制 | $0 | 每次全量重新生成 $3-8 |
| 基线对比 | 规则引擎输出作为 AI 质量的基线 | — |
| 快速原型 | 5 分钟搭好 demo 给利益方看 | 等 API 返回，如果遇到限流更慢 |

**核心洞察**：规则引擎的价值不在于输出质量，而在于**系统韧性**和**开发效率**。它确保即使 AI 模块完全瘫痪，系统仍然可以产出一份"能用但不完美"的词库——而不是彻底无输出。

### 规则引擎的实现策略

以语块生成器为例：

```python
# rule_engine 模式：从预定义模式匹配生成
# 输入：word="kind", pos="adj.", definition="友好的"
# 逻辑：adj. → 尝试 "be + word + to sb." 模式 → "be kind to sb."

# AI 模式：调用 Claude API，从 Prompt 生成
# 输入同上，但 Prompt 包含 standards.yaml 中的正反例
# 输出由 AI 生成，更自然、更多样，但有不可控性
```

规则引擎的输出质量大约在"可用但不够好"水平——语块是高频的但可能不够自然，例句语法正确但可能过于机械。这对调试和原型足够了，正式生产则应使用 AI 模型。

### 优缺点总结

| 维度 | 优点 | 缺点 |
|------|------|------|
| 开发 | 调试快、不花钱 | 需要维护两套生成逻辑 |
| 运维 | API 挂了有兜底 | 规则引擎输出质量不够正式生产 |
| 质量 | 可以作为 AI 输出的基线 | 规则引擎的"天花板"远低于 AI |

## 7. 数据库选型决策

### 演进路径：SQLite → SQLAlchemy + 可选 PostgreSQL

#### 第一阶段：SQLite（项目启动）

**为什么从 SQLite 开始**

| 理由 | 说明 |
|------|------|
| 零配置 | 不需要安装数据库服务器，`data/vocabulary.db` 一个文件搞定 |
| 便携性 | 整个数据库一个文件，可以复制、备份、用 DB Browser 直接查看 |
| 数据规模适配 | 2557 词 × 9 张表，总数据量 < 1MB，SQLite 性能绰绰有余 |
| 团队规模 | 1-3 人，单用户并发，SQLite 无需考虑连接管理 |

#### 第二阶段：SQLAlchemy ORM（当前）

**迁移的直接触发因素**

1. **数据库无关性需求**：产品方希望未来部署到云数据库（PostgreSQL），但当前阶段不想马上迁移。需要一层抽象，使得代码不绑定在 SQLite 方言上。

2. **代码维护性问题**：原始代码有 134 行 DDL 字符串 + 24 处 raw SQL，修改任何表结构都要改两处（DDL + 查询语句）。ORM 把 schema 集中在 `models.py` 中，查询从 SQL 字符串变成 Python 方法调用。

3. **N+1 查询性能问题**：`export.py` 导出时，对每个词做 4 次查询（音标、义项、来源、内容），2557 词 = ~10000 次查询。ORM 的 `joinedload` 一行代码解决，降到 ~4 次查询。

**迁移方法**

| 模式 | 处理方式 |
|------|---------|
| `INSERT OR IGNORE` | 先查后写（`session.query().filter().first()`，存在则跳过） |
| `ON CONFLICT DO UPDATE` | 先查后写（存在则更新字段） |
| `datetime('now')` | `datetime.utcnow()`（Python 侧生成，跨库兼容） |
| `PRAGMA foreign_keys=ON` | SQLAlchemy event listener，仅 SQLite 时启用 |
| `134 行 DDL 字符串` | `Base.metadata.create_all(engine)` 一行替代 |

#### 为什么不引入 Alembic

Alembic 是 SQLAlchemy 的迁移工具，相当于 Django 的 migration 系统。

**不引入的原因**：
- 当前 schema 稳定——9 张表已定义完毕，短期内不会新增字段
- 项目规模小——即使需要改 schema，`Base.metadata.create_all()` + 手动数据迁移的成本也很低
- 团队只有 1-3 人——Alembic 的主要价值在多人协作时保证 migration 的一致性

**引入时机**：当团队 > 3 人，或需要在生产环境做在线 schema 变更时，零成本引入 Alembic。

#### PostgreSQL 的定位

**不是替代 SQLite，而是可选的生产部署方案。**

`DATABASE_URL` 环境变量控制：
- 不设环境变量 → 默认 SQLite（本地开发、个人使用）
- 设为 PostgreSQL 连接串 → 云数据库（团队协作、生产部署）

```bash
# 本地开发（默认）
python cli.py import-excel

# 生产部署
export DATABASE_URL="postgresql://user:pass@host:5432/vocab"
python cli.py import-excel
```

**PostgreSQL 带来的增量价值**：

| 维度 | SQLite | PostgreSQL |
|------|--------|-----------|
| 并发 | 单写入者，写锁粒度大 | 多连接并发读写，MVCC |
| 远程访问 | 文件锁，只能本地 | TCP 连接，支持远程 |
| 数据完整性 | 类型宽松（TEXT 可存任何东西） | 严格类型检查 |
| 全文搜索 | 需手动实现 | 内置 `tsvector` + GIN 索引 |
| 备份恢复 | 复制文件 | `pg_dump` / 时间点恢复 |
| 扩展性 | 单文件上限约 140TB（理论），实际 GB 级后性能下降 | 单表可达 TB 级 |

**当前阶段 PostgreSQL 非必须**：数据量 < 1MB，用户数 < 5 人，SQLite 完全胜任。但架构上预留了切换能力，需要时一个环境变量即可切换。

## 8. 标准前置的落地方法论

### standards.yaml 的设计哲学

这个文件要同时服务三类读者：
1. **学科产品**：看正反例，理解"什么是好的语块"
2. **AI 模型**：从 Prompt 模板中获取约束
3. **规则引擎**：读取约束参数（`must_contain_target_word: true`）

**文件结构**

```yaml
content:
  chunk:
    definition: "..."        # 给学科产品看的定义
    constraints:             # 给规则引擎和 AI 共用的约束
      must_contain_target_word: true
      min_words: 2
      max_words: 6
    positive_examples:       # 给学科产品看正例，给 AI 做 few-shot
      - word: "kind"
        pos: "adj."
        good: "be kind to sb."
        reason: "高频搭配，包含介词结构"
    negative_examples:       # 给学科产品看反例，给 AI 做反面教材
      - word: "kind"
        bad: "kind person"
        reason: "过于简单，没有介词结构"
    evaluation:              # 给 AI 审核器的评分标准
      criteria:
        - name: "自然度"
          weight: 0.4
          pass_threshold: 3
```

### 正反例的关键作用

正反例不是装饰性的——它们是 few-shot learning 的核心素材。

**AI 生成时**：Prompt 模板中嵌入正反例，引导模型理解"什么是好语块"。实验表明，加了 3 个正例 + 2 个反例的 Prompt 比无例子的 Prompt，语块质量提升约 30%（基于人工评估）。

**规则引擎时**：正反例帮助开发者理解业务意图，写出更准确的规则。例如，看到"a dog"被标为反例后，开发者就知道规则引擎应该过滤掉"冠词+名词"模式。

**AI 审核时**：评分标准 + 正反例构成审核的 few-shot context。审核器不是凭空判断"这个语块好不好"，而是参照标准中的正反例做对比。

### 标准层的当前局限

**诚实评估：standards.yaml 目前是"框架已搭好，内容不够满"。**

| 维度 | 正例数量 | 反例数量 | 评估 |
|------|---------|---------|------|
| 语块 | 3 | 3 | 勉强够用，但需要更多边界案例 |
| 例句 | 2 | 2 | 不够——需要覆盖更多义项类型 |
| 助记 | 2 | 2 | 不够——需要覆盖 5 种助记方法的示例 |

**建议：每个维度至少需要 10 个正例 + 5 个反例才能充分指导 AI 和学科产品。当前差距较大。**

## 9. 词包体系的产品设计

### 设计理念：中央内容池 + 选择视图

```
中央数据库（所有单词、义项、内容的共享池）
    |
    |-- 词包 A（人教版中小学）：2819 个义项
    |-- 词包 B（高中英语）：3500 个义项
    +-- 词包 C（商务英语）：2000 个义项

good "好的" 的语块 "be good at" → 词包 A 和 B 共享
```

### 为什么词包不是独立数据库

**被否决的方案**：每个词包一个独立数据库，各自管理自己的单词和内容。

否决原因：
- 同一个单词在多个词包中出现（如 `good` 可能同时在中小学包和高中包中），独立数据库意味着**同一条内容需要生成两次、审核两次**
- 修改了 `good` 的语块后，需要在每个数据库中同步修改
- 数据不一致风险极高

**我们的方案**：中央内容池 + 词包是"选择视图"。

```
packages 表：定义词包的名称、目标受众
package_meanings 表：每条记录表示"词包 X 包含义项 Y"
```

导出时按词包过滤：

```sql
SELECT ... FROM words
JOIN meanings ON ...
JOIN package_meanings ON meanings.id = package_meanings.meaning_id
WHERE package_meanings.package_id = ?
```

### 优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 内容复用 | 同一条语块/例句只生成一次、审核一次 | 词包之间有耦合——修改内容会影响所有包含该义项的词包 |
| 一致性 | 多个词包中同一个词的内容保证一致 | 如果不同词包需要"同一个词的不同语块"（如中小学用简单语块、高中用复杂语块），当前架构不支持 |
| 运维 | 新增词包只需添加 package_meanings 记录，无需重新生成 | — |

**当前不需要"词包级内容差异化"**——因为只有一个词包。如果未来确实需要"同一个词在不同词包中用不同的语块"，可以在 content_items 表加 `package_id` 列来扩展。

| 审核 8000 条内容 | 另一位学科产品 | ~200 小时 |
| 每学期更新 | 学科产品 | ~50 小时 |
| **年化总成本** | | **700+ 小时** |

#### 纯 AI 方案（无人工审核）

| 阶段 | 方式 | 单次成本 |
|------|------|---------|
| 内容生成 | Claude Sonnet | $5-15 |
| AI 审核 | 自动 | $2-5 |
| 人工抽检 | 学科产品 | ~5 小时 |
| **年化总成本** | | **$14-40 + 10h 人力** |

成本最低，但质量风险最高。5% 的错误率意味着 ~400 条有问题的内容直接到达学生。

### ROI 总结

当前方案在 **年化 $20-258 + 40-90h 人力** 的成本下，实现了：
- 与全人工方案相当的内容质量（人工审核兜底）
- 全人工方案 10% 的人力投入（600 小时 → 60 小时）
- 比纯 AI 方案更高的质量保障

**关键假设**：这个 ROI 计算假设 AI 审核层能过滤掉 60-80% 的语义问题。如果 AI 审核的实际效果不及预期，人工审核的工作量会相应增加。

## 11. 诚实的局限性评估

### 架构层面

| 局限 | 影响 | 应对策略 |
|------|------|---------|
| **CLI-only 界面** | 学科产品无法自主审核，依赖 AI 产品经理代操作 | P2 加 Web UI（推荐 Streamlit） |
| **单机部署** | 无法多人同时操作（SQLite 写锁） | 切换到 PostgreSQL 后解决 |
| **无自动化流水线** | 每个步骤需要手动执行 CLI 命令 | 可加 Makefile 或 CI/CD pipeline 串联 |
| **无版本回滚** | 重新生成内容会覆盖旧版本 | `audit_log` 记录了 old_value，但恢复需要手动操作 |

### 数据层面

| 局限 | 影响 | 根因 |
|------|------|------|
| **57 个词缺音标** | 这些词的音标字段为空，Gate 1 标记为错误 | `eng_to_ipa` 库无法处理复合短语和专有名词（如 "P.E."、"Good morning"） |
| **120 个词释义措辞不统一** | 源数据中同一义项有不同的措辞（如"手臂"vs"胳膊"），导致本应合并的义项被当作两个独立义项 | 上游问题——12 份教材词表没有做到 100% 使用同一词典的表述 |
| **29 条例句缺中文翻译** | 规则校验已标记，但未修复 | 规则引擎生成的例句部分缺翻译，需要补全 |

### 质量层面

| 局限 | 影响 | 优先级 |
|------|------|--------|
| **6 个 AI 审核器全未实现** | 语义层质量无法自动检测，全靠人工 | P0（但被学科产品出标准阻断） |
| **规则引擎输出质量有限** | "be + adj. + to sb."这类模式虽然语法正确，但覆盖面有限 | 已知，接受为 tradeoff |
| **导出门禁过于严格** | 8000 条中有任何一条未 approved 就不允许导出 | 有 `--force` 旁路，但正式发布时确实需要全量审核 |

### 流程层面

| 局限 | 影响 | 解决方案 |
|------|------|---------|
| **标准层内容不完整** | 正反例不够（每维度 2-3 个，理想是 10+5 个），AI 生成和审核的效果受限 | 催促学科产品补全 |
| **无自动重跑机制** | Gate 2 的"自动重试 3 次"在 standards.yaml 中定义了但未实现 | 需要开发 |
| **审核工作流单一** | 只有 approve/reject，没有"部分修改后通过" | 可加 "approve with edit" 操作 |

## 13. 决策速查表

一句话回答"为什么"：

| 问题 | 回答 |
|------|------|
| 为什么不直接人工编写所有内容？ | 400 小时人力 vs 15 秒机器 + 60 小时审核，ROI 不合理 |
| 为什么不完全信任 AI 输出？ | 教育内容零容错，当前 AI 在细粒度语义判断上准确率 ~90%，不够 |
| 为什么用 YAML 而不是 Word 文档？ | 机器可读——规则引擎和 Prompt 可以直接读取约束参数 |
| 为什么内容挂在义项上不挂在单词上？ | 防止多义词张冠李戴——"be good at" 是形容词的语块，不是名词"货物"的 |
| 为什么助记挂在单词上不挂在义项上？ | 助记面向拼写/发音，与义项无关——"kind 和 king 只差一个字母"适用于所有义项 |
| 为什么导出门禁默认阻断？ | fail-safe——宁可导出失败，不可把未审核内容发给学生 |
| 为什么保留规则引擎？ | 系统韧性——API 挂了有兜底，调试时零成本 |
| 为什么从 raw SQL 迁移到 ORM？ | 数据库无关性 + N+1 优化 + 代码可维护性 |
| 为什么不引入 Alembic？ | 当前 schema 稳定、团队小、零成本可后加 |
| 为什么 ID 从 1001 起？ | 避免与前端已有 ID 冲突 |
| 为什么 CLI 而不是 Web UI？ | 开发成本 2 天 vs 4 周，当前用户是开发者不是学科产品 |
| 为什么词包是"选择视图"？ | 共享内容池——同一条语块在多个词包间复用，不重复生成 |
| 为什么四层架构而不是三层？ | 反映内容成熟度——标准→原料→半成品→成品，而非请求流向 |
| 为什么先查后写而不用 UPSERT？ | 跨库兼容——SQLite 和 PostgreSQL 的 UPSERT 语法不同 |
| 为什么不做微服务？ | 团队 1-3 人、数据 <1MB，单体的简单性远超微服务的灵活性 |

---

> **最后一句话**：这个系统的所有设计决策都指向一个目标——**让人做人该做的事（学科判断），让机器做机器该做的事（规模化生产 + 规则化过滤）**。任何看起来"过度设计"的地方，本质上都是在保护那条质量红线：学生拿到的每一条数据都必须是对的。
